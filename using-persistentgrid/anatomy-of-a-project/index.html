Importing the grid

There are three SCSS preset that can be used as a template for your projects:

  • basix
  • framework
  • framework-extended

The preset framework-extension is still under constrution.

Basic
// Import Compass, no-game without it
@import "Compass";

// Import the settings: first the defaults and then the skin settings;
@import "persistentgrid";
The basics: a simple but flexible grid system.
Framework
// Import Compass, no-game without it
@import "Compass";

// Import the settings: first the defaults and then the skin settings;
@import "framework/helpers";

// Import the settings: first the defaults and then the skin settings;
@import "framework/settings";
@import "skin/settings";

// Import the behavior, this is where the styles will be rendered
@import "framework/behavior";
@import "skin/behavior";
The basics: a simple but flexible grid system.

The directory structure is designed such, that it will allow for easy implementation of new modules and styles. On a lower level, it is also possible to extend existing styles. This can come in handy if - for example - you want to use the existing buttons, but only add more rounded corners. An example setup of this can be found in the UI/Buttons module.

The basic structure for any module is:

  • module
    • _all.scss
    • _behavior.scss
    • _settings.scss
    • _helpers.scss
    • helpers
      • _functions.scss
      • _mixins.scss

The module root contains a base file that you will call via the @import directive. See the section 'Configuring PG Framework' for more information.

A module may also be part of a group, for instance 'Buttons' is a module that sits within the 'UI' group:

  • /Groupname
    • /ModuleOne
      • /base
      • /skins
      • _all.scss
      • _moduleone.scss
    • /ModuleTwo
      • /base
      • /skins
      • _all.scss
      • _moduletwo.scss

Workflow of a new FE project

Configuring PG Framework

As all the modules contain their individual settings files, and some of them inherit settings from other modules (for example, the Vertical Rhythm module inherits its base-font-size and base-line-height from the Type module), it is important to load all settings in the correct order. Also - and this very importany - all settings must be loaded before any behavior is rendered. This is especially important when extending existing behavior.

For each stylesheet that you need rendered, you import resources in this order:

  1. Dependencies:
    1. @import "compass";
    2. @import "Skin/base";
    3. @import "Helpers/base";
  2. Base modules:
    1. @import "Reset/base";
    2. @import "Type/base";
    3. @import "Baseline/base";
    4. @import "Base/base";
    5. @import "PersistentGrid/base";
  3. Modules:
    1. @import "Forms/base";
    2. @import "UI/Buttons/base";
    3. @import "UI/Linkbar/base";
    4. @import "Lists/Lists/base";

First, you import Compass, as PG Framework is dependent on a number of Compass modules. You can of course also import individual Compass modules, but I prefer to just include everything at once.

Second, you import your skin. This is important, as all settings are imported through the Skin module before any of the required modules are rendered.

Nexr, you call the helper files. (These may be moved into the skin folder).

Now, the base modules can be called. The base modules are Reset, Type, Vertical Rhythm, Base and Persistent Grid.

Last, you import the necessary modules - in any order.

You can mix and match the modules in any way that fits your project. For example, you may create a base stylesheet that contains the base modules and additional stylesheets that contain only those modules that are needed per pageview.
Or, you create different stylesheets for different skins by importing alternate skins.
